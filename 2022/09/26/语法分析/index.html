语法分析设计

语法分析综述

在前面的词法分析的基础上，考虑到实现后续功能的便捷性，我们在此处建立了一棵语法树。

当然，在使用递归下降的方法进行语法分析的要求是文法当中不存在左递归，同时尽可能的避免回溯。下面对这些情况进行考虑。

左递归

考虑到所提供的部分文法当中存在左递归，例如文法AddExp -> MulExp | AddExp ('+' | '−') MulExp中，可采用转换为扩展的 BNF 范式形式的方法予以解决，我们可以将其改写成AddExp -> MulExp { ('+' | '−') MulExp }，即可避免左递归的问题。

但是由于我们改写了文法，相应的语法树也会有所改变。

比如上面这种情况，语法树的层数发生了改变，导致语法分析阶段输出结果的时候会少输出一些 <AddExp>，可以通过调整输出 <AddExp> 的位置来解决。

也就是说，把输出 <AddExp> 的句子放在每个mulExpNode.print()的后面即可。

回溯问题

回溯问题的存在是因为对于某个非终结符号的规则其右部有多个选择，且其first集相交，从而导致分析到此语法成分时，不能仅根据当前词法成分判断接下来的语法成分是什么。

例如VarDef -> Ident { '[' ConstExp ']' } | Ident { '[' ConstExp ']' } '=' InitVal就出现了回溯问题，我们想要避免这个回溯问题也很简单，只需要改写文法使之变成VarDef -> Ident { '[' ConstExp ']' } [ '=' InitVal ]即可解决回溯问题。

类似的还有 Stmt 文法。但是考虑到 Stmt 文法比较复杂，我们在后面拿出专门的一节再来说这个问题。

语法分析文件结构

每个部分作用

node 文件夹

在 node 文件夹内是所有的非终结符号作为的树结点。

以文法CompUnit -> {Decl} {FuncDef} MainFuncDef为例。

{Decl}表示Decl有可能有零个到多个，我们选择使用List<DeclNode>来存储所有的 Decl 的树结点。类似的使用List<FuncDefNode>来存储所有的 FuncDefNode 的树结点。

Parser

文件结构如下所示。

Map<NodeType, String> nodeType的作用是将要输出的语法成分的名字放在一个 Map 里面，直接使用nodeType.get()方法可以方便的进行输出。同时如果以后需要修改输出的语法成分的名字等，也只需要修改这一个地方即可完成。

Token match(TokenType tokenType)是判断当前 tokens 列表当中下标为 index 的元素的 type 是否与所期望的 tokenType 相同。如果相同则表示这个语法成分是正确的，否则就给出 RuntimeException，同时输出相关错误信息。

printParseAns()即为输出语法分析阶段答案的方法，通过遍历语法树的方式进行答案的输出。
